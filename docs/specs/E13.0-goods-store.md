# E13.0 — One Goods store

Feature spec for sub-epic E13.0 (milestone M3 — Guilds & obligations, [PRD](../PRD.md));
a foundation E13 rides on, sequenced **before** #100. Terms per
[CONTEXT.md](../../CONTEXT.md). Grilled and decided with the owner on 2026-07-19.
Status: **draft**.

Grill inputs: Professor findings **F4** (five hand-maintained "what is a site" enumerations;
the netWorth array is the silent one) and **F7** (deliver addressing is inexpressible intent),
[design-notes/professor-construction-review.md](../design-notes/professor-construction-review.md);
the owner-led E13 grill of 2026-07-19, which reopened the site-registry decision locked
earlier the same day; Engineer ("Carl") Tech pass, same date.

Scope in one line: every place goods can sit becomes **one encapsulated Goods store** with a
derived policy and one Transfer primitive — a **behavior-preserving refactor of the four
stores that exist today**, shipping no new gameplay.

Explicit non-goals: **explicit addressing** (E13 — it is a behavior change and would blow
this epic's cover); the Storehouse itself and processing plants (E13/E15); unifying
`market↔hold` with the Transfer primitive (deferred — see §Tech, Transfer); the store's
**direction** rules, i.e. who may put goods in and whether withdrawal is allowed (contract in
ADR-0008, implemented when they gain consumers in E13/E15); per-lot receipt time (rejected,
see below); the four *loud* site enumerations — tick phase order, the rush command trio, the
deliver chain's precedence, UI section branches — which stay hand-maintained by deliberate
decision; Professor F5 and F6, which stay live watches.

## Design

The player sees nothing. That is the acceptance criterion, not a caveat — this sub-epic
exists so that E13 and E15 can be built on a model the game had outgrown, and its entire
justification is that behavior does not change.

### Naming the concept, not building machinery

The debt this repays is not missing machinery — it is a **missing name**. #99 extracted a
shared construction-site engine because a second caller appeared, but left three parallel
state holders behind it; the code was deduplicated while the concept was not. "A place where
goods can sit" was never a term, so each new mechanic re-derived from scratch whether it was
a site, whether netWorth saw it, and what shape it took. Three sessions were spent on that
re-derivation before it was named.

The operational lesson, recorded because it generalizes: **when you extract a shared engine
because a second caller appeared, ask whether the *concept* needs unifying, not just the
code.** Deduplication without naming leaves exactly this debt.

### The rule the model enforces

Company value changes **only** through a booked Ledger event (CONTEXT.md, Ledger §Value law).
Goods you own count wherever they lie — Cargo, construction site, Building; Ships and
Buildings themselves carry no book value, so the company-value chart keeps telling the honest
story (a build is a visible dip, then steeper growth).

Stated from the player's side, this is why it matters: the company-value chart is the only
instrument on which "buy low, store, sell after the drift" — E13's whole third dimension of
pillar 2 — can be read at all. If a store were omitted, storing goods would read as burning
money and withdrawing them as creating it, which inverts the mechanic. The failure would be
silent: no error, no crash, no in-game symptom.

### Why an invariant and not a registry

The rejected alternative was a typed **site registry**: a discriminated union over site kinds
with an exhaustive switch, consumed by netWorth. It was rejected for two reasons. It guards a
set that only happens to be closed at three — E15's plants and E13's Storehouses arrive as
*collections*, which a loop already covers, so the guard protects a door nobody walks
through. And it would oblige every future author to ask "am I a site kind? must I register?",
reproducing the discussion it was meant to end.

The guard is instead a **value-neutrality invariant**: moving your own goods never changes
company value. It enumerates nothing, so it needs no update when a place is added.

**Stated honestly, because it must not be discovered later:** this does not eliminate
enumeration. `companyStores` is still hand-maintained, and the invariant bites only because
the test generates its moves from that enumeration. What it buys is the conversion of a
**silent** failure into a **loud** one — a store missing from `companyStores` is a store the
player cannot fill, visible in the first minutes of play, where the netWorth omission failed
silently and forever. That conversion was F4's actual goal.

### Per-lot receipt time: deliberately not built

Tracking when goods entered a store (for future spoilage, storage fees or FIFO valuation) has
**no consumer today** — E15 defers ice decay to a later epic. Building it now would be
generalization ahead of a mechanic that has not been designed.

The insurance instead is encapsulation, with a hard acceptance criterion:

> **Adding a property to goods held in a store must touch ONLY: the `GoodsStore` type, the
> three accessor functions, and the save migration. If it touches anything more, this
> sub-epic did not do its job.**

This is tested, not asserted — see §Testing, C5.

## Tech

Engineer pass by "Carl", 2026-07-19, against `main @ 03bcdf4` with a verified green baseline
(611 unit tests, 41 files).

### The store — `src/sim/goodsStore.ts` (new)

`GoodsStore` is an **opaque alias over today's on-disk `Record<GoodId, number>`**: same JSON,
same runtime object, but the type forbids every consumer outside the module from touching
contents.

```ts
declare const HELD: unique symbol;
export type GoodsStore = { readonly [HELD]: true };

export function emptyStore(): GoodsStore;
export function storeOf(partial: Partial<Record<GoodId, number>>): GoodsStore; // zero-fills in GOOD_IDS order
export function amountOf(store: GoodsStore, good: GoodId): number;
export function withAdded(store: GoodsStore, good: GoodId, qty: number): GoodsStore;
export function withRemoved(store: GoodsStore, good: GoodId, qty: number): GoodsStore;
export function totalHeld(store: GoodsStore): number;   // the cargoUsed fold, generalized
```

Feasibility was verified against this repo's own `tsc` before adoption, with three
`@ts-expect-error` fixtures — dotted access, dynamic access, raw-literal assignment. All three
compiled clean, which (an *unused* `@ts-expect-error` being itself an error) proves each is a
compile error outside the module, while accessor code and `JSON.stringify` round-trips
compile fine. `tsconfig.app.json` sets `noEmit` with no declaration emit, so the usual
"exported type uses private name" objection to `unique symbol` brands does not apply here.

Thin wrappers keep the diff small and barrel exports intact: `emptyCargo()` → `emptyStore()`,
`emptySiteStore()` → `emptyStore()`, `cargoUsed(ship)` → `totalHeld(ship.cargo)`.

### Policy — discriminated union, derived, never serialized

```ts
export type StorePolicy =
  | { readonly kind: "hold"; readonly capacity: number }                 // capacity = Ship.hold, passed in
  | { readonly kind: "constructionSite"; readonly recipe: Record<GoodId, number> };
  // E13: | { kind: "storehouse"; filter: readonly GoodId[]; capacity: number }
  // E15: | { kind: "plantInput"; chain: ChainId; capacity: number }
  //      | { kind: "plantOutput"; good: GoodId; capacity: number }

export function accepts(store: GoodsStore, policy: StorePolicy, good: GoodId, qty: number): number;
```

`accepts` is where the goods filter, the capacity clamp and the per-good remaining need all
live — one place, per the store-owns-its-policy rule. A function table is disqualified
outright: a stored closure does not survive ADR-0004's `JSON.stringify`. The policy is derived
at the point of use, following the grain the repo already set (`shipyard.ts:159-163, 256-265`
recompute recipes live rather than storing them). `noFallthroughCasesInSwitch` plus an
exhaustive `never` check makes an unhandled kind a compile error.

**This is not the rejected site registry** (owner ruling, OQ4): one union consumed by one
function answering "what do I accept", versus a registry consumed by five subsystems as the
guard against a silent failure.

**A moving store does not differ from a fixed one.** The Hold's *policy* differs (flat total
capacity rather than a per-good recipe) and its *addressing* differs (`shipId` rather than an
owner path), but the type and the three accessors are identical. Movement belongs to the Ship.

### Transfer — `src/sim/transfer.ts` (new)

```ts
export type StoreRef =
  | { readonly kind: "hold"; readonly shipId: ShipId }
  | { readonly kind: "hqBuild" }
  | { readonly kind: "shipyardBuild" }
  | { readonly kind: "refit" };
  // E13: | { kind: "storehouse"; portId } ;  E15: | { kind: "plantInput" | "plantOutput"; portId }

export function companyStores(world: World): readonly StoreRef[];
export function readStore(world: World, ref: StoreRef): GoodsStore | null;
export function policyFor(world: World, ref: StoreRef): StorePolicy | null;
export function writeStore(world: World, ref: StoreRef, next: GoodsStore): World;
export function moveOwnGoods(world: World, from: StoreRef, to: StoreRef, good: GoodId, qty: number | "max"): World;
```

`writeStore` is the single home for the four write closures threaded through `tryDeliver`
today (`commands.ts:156-182, 359-415`), and the one place `withShipyard` (`shipyard.ts:148-153`)
is called per site instead of ten times.

**Scope call (owner, OQ1): E13.0 implements only the thaler-free half** — `hold↔site`.
`market↔hold` stays on the existing `applyTrade` seam (`commands.ts:746-798`), because that
path also carries the price walk, the purse, market-impact stock mutation and contract
fulfilment attribution; folding it in is the one part of the collapse that risks the
behavior-preservation guarantee. The `StoreRef` vocabulary is defined for all four pairs
regardless, so the remaining collapse stays a cheap later step.

### The delivery chain — extracted, not changed

**Load-bearing for the E13.0/E13 split; belongs in the acceptance criteria verbatim.** The
`deliver` guard-clause chain (`commands.ts:337-439`) becomes one pure function

```ts
function resolveDeliveryTarget(world: World, ship: Ship, good: GoodId): StoreRef | null
```

with `deliver` becoming `resolve → moveOwnGoods(hold → target)`. Same precedence
(HQ → Shipyard site → Refit), same fall-through, same zero-need completion resolution
(`commands.ts:435-437`). This makes addressing **representable but not player-selectable**:
E13 then *deletes* `resolveDeliveryTarget` and lets the player name the `StoreRef` — a
one-function deletion instead of chain surgery. If E13.0 instead hardcodes the chain inside
the primitive, E13 pays for the split.

### Persistence and determinism

**No `SAVE_VERSION` bump. Stays at 13** (owner ruling, OQ6). The runtime shape is byte-identical
— `GoodsStore` is a compile-time alias over the same `Record<GoodId, number>` that ships
today; no field is added, removed or renamed. `READABLE_VERSIONS` stays `{12, 13}`;
`migrateV12ToV13` (`persistence.ts:84-90`) is untouched; no migration is written. Per-store
migration path: **none required**.

Hazard this deliberately avoids, recorded so a later bump does not walk into it:
`e2e/fixtures/ledger-scenario.json` is version **12** and is loaded by `e2e/ledger.spec.ts:55`.
Under the one-step-migration precedent, any bump to 14 sets `READABLE_VERSIONS = {13, 14}`,
that fixture stops parsing and the Ledger E2E specs fail. **Any future bump must regenerate
that fixture in the same PR.**

Determinism (ADR-0003):

1. **Key-order dependence.** `withAdded` uses object spread, so a store parsed from JSON
   inherits the file's key order. This cannot affect sim math — there is zero
   `Object.keys`/`entries`/`values` in production code across `src/sim`, `src/store`, `src/ui`;
   every fold walks `GOOD_IDS` (`goods.ts:17-23`). It *can* affect raw `JSON.stringify`
   comparison, so **the golden digest walks `GOOD_IDS` explicitly** rather than stringifying
   the world. `emptyStore`/`storeOf` fill in `GOOD_IDS` order, so fresh stores are canonical.
2. **Fold order in `totalHeld`** is fixed to `GOOD_IDS`. Quantities are integers everywhere
   (`commands.ts:612` validates), so the fold is exact regardless; fixing the order keeps it
   exact if a non-integer good ever appears.
3. **No RNG involvement.** Nothing here draws; the store is downstream of every RNG phase.
4. **`companyStores` order is not load-bearing for netWorth** (a sum) but **is** load-bearing
   the moment a caller spends from the shared purse while walking it. E13.0 must **not**
   re-route the tick's auto-draw through `companyStores`: the runners keep their explicit
   HQ → Shipyard → Refit order (`tick.ts:452-461`, Professor F3). Pinned by C4.

**Restated acceptance shape (owner ruling, OQ3).** "The entire existing suite passes
unchanged" is *not* the criterion and cannot be: encapsulation stops roughly 72 test index
sites and 33 literal constructions from compiling. Leaving tests indexing contents directly
would make the acceptance criterion false exactly where it is later tested. The criterion is
therefore: **no test's expected values change — only the syntax of access — and C1's golden
digest is the behavior-preservation proof.** Test constructions get shorter
(`storeOf({ grain: 5 })` beats a five-key literal).

## Issue cut

Strict dependency order. E13.0-2 merging **before** the refactor is what makes E13.0-3's diff
safe to review at speed: the reviewer reads the digest's green as the behavior argument and
the rest as mechanics.

| # | Track | Scope | Depends on | Size |
| --- | --- | --- | --- | --- |
| E13.0-1 | docs | CONTEXT.md entries + Ledger Value law; ADR-0008; this spec. **First — glossary-first law** (SELFCHECK §4.4) | — | Small |
| E13.0-2 | tests | C1 golden-run digest fixture + C4 phase-order snapshot, **written and merged against `main`, green, before any refactor** | E13.0-1 | Medium |
| E13.0-3 | sim + ui | `goodsStore.ts`, `goodsStorePolicy.ts`, `transfer.ts`; migrate the four stores; `resolveDeliveryTarget` extraction; `computeNetWorth` over `companyStores`; the ~127-site rewrite; `goodsStore.typeguard.ts`; the C3 and C5b drills reported in the PR body | E13.0-2 | **Large** — tier 3 review |

The sim/ui split does not survive contact with the opaque type: the moment `Ship.cargo`
becomes `GoodsStore`, `src/ui` and `src/store` stop compiling, so a sim-only PR would be red
on `npm run typecheck` — a merge gate. E13.0-3 is therefore one PR across both.
